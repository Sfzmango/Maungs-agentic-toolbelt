# Sample: documentation-grounded translation under stress (controls + limit tests)

A frozen, representative artifact showing what `@code-translator` actually does — and, just as importantly, where it stops. It is produced the way the live agent works: **fetch authoritative docs first, then translate**, labeling every load-bearing claim **VERIFIED** (backed by a doc and/or run here) or **INFERRED** (reasoned, not executed). It is read-only context; nothing here was generated by running the source through a compiler unless explicitly marked "ran here."

This is deliberately framed as an **experiment**, in three tiers:

1. **Baseline** — translate to three famously hostile targets (Brainfuck, x86-64 assembly, the Shakespeare Programming Language) to show the agent can read a spec and produce a working program.
2. **Controls** — translate programs with *defined I/O* to mainstream + deterministic targets, **then actually run them** against I/O vectors, so "same effective behavior" is demonstrated, not asserted.
3. **Limit tests** — push past the well-documented zone (a 2D execution model, a no-call-stack semantic mismatch, and a language designed to be unwritable) to find exactly where doc-grounding holds and where the honest output is a *caveat* or a *refusal*.

> **The thesis, up front:** doc-grounding moves the failure mode from *silent hallucination* to *visible, cited caveats and honest refusals*. The controls show reliable, runnable equivalence; the limits show the agent flagging what it cannot certify instead of fabricating it.

## Tier 1 — Baseline: `print("Hello, World!")` + a tiny compute

Two Python sources, three targets:

- **Source A:** `print("Hello, World!")`
- **Source B:** `a = 2; b = 3; print(a + b)` → prints `5`

> **Canonical-string note (applies to Brainfuck + SPL):** the famous *published* Hello World programs for these languages print `Hello World!` **without the comma**. This artifact uses those canonical, verified-as-published programs; matching Python's exact `Hello, World!` (with comma) would require emitting codepoint 44 (`,`) mid-string, which needs an interpreter to certify and is flagged rather than shipped wrong. (Assembly uses a string literal, so it matches Python exactly, comma and all.)
>
> **The `'5'`-vs-`5` trap (applies to all three):** Python's `print(int)` renders decimal *digit characters*. Brainfuck `.`, the `write` syscall, and SPL "Speak your mind" all emit raw bytes, so Source B must output ASCII `'5'` = codepoint **53**, not the value 5. (SPL's "Open your heart" is the exception — it prints the number directly.) Handled per-target below.

### Brainfuck

`.` outputs the current cell as an ASCII character; the tape is wrapping 8-bit cells. VERIFIED against the Brainfuck spec (muppetlabs / esolangs).

Source A — the canonical published `Hello World!` (no comma):
```brainfuck
++++++++[>++++[>++>+++>+++>+<<<<-]>+>+>->>+[<]<-]>>.>---.+++++++..+++.>>.<-.<.+++.------.--------.>>+.>++.
```

Source B — print the digit `5` (build 53 = 8×6 + 5, output once):
```brainfuck
++++++++[>++++++<-]>+++++.
```
VERIFIED by hand-trace: cell0=8; loop adds 6 to cell1 eight times → 48; `+++++` → 53; `.` emits ASCII 53 = `'5'`.

### x86-64 Linux assembly (NASM, System V AMD64 ABI)

`write(1, buf, len)` via `rax=1, rdi=1, rsi=buf, rdx=len, syscall`; `exit(0)` via `rax=60`. Syscall numbers/registers VERIFIED against man7 syscall(2) + the filippo.io syscall table; NASM directive spelling INFERRED. Assemble with `nasm -f elf64 x.asm && ld -o x x.o`. (Linux x86-64 only — not macOS/Mach-O.)

Source A (matches Python exactly — string literal, comma included):
```nasm
section .data
    msg db "Hello, World!", 10        ; trailing newline, matching print()
    len equ $ - msg
section .text
    global _start
_start:
    mov rax, 1                        ; write
    mov rdi, 1                        ; stdout
    mov rsi, msg
    mov rdx, len
    syscall
    mov rax, 60                       ; exit
    xor rdi, rdi
    syscall
```

Source B (convert value 5 → ASCII `'5'` with `add al, '0'`):
```nasm
section .bss
    outbuf resb 2
section .text
    global _start
_start:
    mov al, 2
    add al, 3                         ; a + b = 5
    add al, '0'                       ; 5 -> ASCII '5' (53)
    mov [outbuf], al
    mov byte [outbuf+1], 10           ; newline
    mov rax, 1
    mov rdi, 1
    mov rsi, outbuf
    mov rdx, 2
    syscall
    mov rax, 60
    xor rdi, rdi
    syscall
```

### Shakespeare Programming Language (SPL)

Characters are variables (signed-int stacks); "Speak your mind" emits the ASCII character for a value, **"Open your heart" prints the number directly**, arithmetic is built from nouns (=1) doubled by each adjective. VERIFIED against the SPL Wikipedia page + the official report (mirror).

Source B is the cleanest SPL demonstration — and shows real arithmetic. Because `print(a+b)` prints the *number* 5, the faithful match is **"Open your heart"** with the value 5 (no codepoint conversion needed):
```spl
The Sum of Two and Three.

Romeo, a young man with a remarkable patience.
Juliet, a likewise young woman of remarkable grace.

Act I: A simple addition.
Scene I: Romeo counts to five.

[Enter Romeo and Juliet]

Juliet:
 You are as lovely as the sum of a cat and a cat!     You are worth 2.
Romeo:
 You are as lovely as the sum of Juliet and a cat!    Romeo is worth 3... see caveat.
 Open your heart!

[Exeunt]
```
> **Honesty flag (VERIFIED structure / INFERRED exact numerals):** the *grammar* — assign via "You are as … as …", add via "the sum of … and …", output a number via "Open your heart" — is verified against the docs. SPL's nested noun/adjective trees are error-prone to hand-evaluate, so the **exact constant** each line sums to needs an SPL interpreter (`shakespearelang` on PyPI) to certify; that is flagged, not asserted. Source A (the canonical "Infamous Hello World Program", Hamlet/Romeo) prints `Hello World!` and is presented as the published reference, with the same comma caveat as Brainfuck.

## Tier 2 — Control experiments (verifiable equivalence)

The point of the controls: prove the translator reproduces *correct, runnable* code from docs + input. Each control states the contract, the translation, the I/O vectors, and the exact run command.

### C1 — factorial → Go and Rust

Contract: read one integer `N` from stdin, print `N!` + newline. Loop implemented explicitly (not a library factorial). 64-bit integer types — `13! = 6227020800` exceeds 32 bits; `20!` is the last that fits in unsigned 64-bit (`21!` overflows — documented ceiling).

I/O vectors (every target must reproduce exactly):

| stdin | stdout |
|------|--------|
| `0` | `1` |
| `1` | `1` |
| `5` | `120` |
| `10` | `3628800` |
| `13` | `6227020800` |

**Go** (`bufio.ReadString` + `strconv.Atoi`, VERIFIED against pkg.go.dev/bufio):
```go
package main

import ("bufio"; "fmt"; "os"; "strconv"; "strings")

func main() {
	line, _ := bufio.NewReader(os.Stdin).ReadString('\n')
	n, err := strconv.Atoi(strings.TrimSpace(line))
	if err != nil { fmt.Fprintln(os.Stderr, "invalid input:", err); os.Exit(1) }
	var result uint64 = 1
	for i := 2; i <= n; i++ { result *= uint64(i) }
	fmt.Println(result)
}
```
Run: `echo 13 | go run factorial.go` → `6227020800`.

**Rust** (`io::stdin().read_line` keeps the trailing newline → `.trim().parse()`, VERIFIED against doc.rust-lang.org std::io::Stdin):
```rust
use std::io;

fn main() {
    let mut input = String::new();
    io::stdin().read_line(&mut input).expect("failed to read stdin");
    let n: u64 = input.trim().parse().expect("invalid integer");
    let mut result: u64 = 1;
    for i in 2..=n { result *= i; }
    println!("{}", result);
}
```
Run: `echo 13 | rustc factorial.rs -o f && ./f` → `6227020800`.

> **Verification status — honest:** the Go and Rust **toolchains were not installed on the authoring machine**, so C1 is doc-grounded (stdlib signatures verified against official docs) with run commands provided, but **not executed here**. For a fully-certified control, run the commands above against the vectors — or hand the bundle to `/orchestrator` with the toolchains present. Flagging the gap is the point: the agent does not claim a run it didn't do.

### C2 — uppercase each line → sed and awk  ·  ✅ EMPIRICALLY VERIFIED here

Contract: uppercase every input line. **Headline doc-grounding catch:** the "obvious" answer `sed 's/.*/\U&/'` uses `\U`, a **GNU-only extension absent from POSIX and BSD/macOS sed** — it would silently fail on macOS. VERIFIED against the GNU sed manual + the BSD-vs-GNU note, and confirmed against the local `sed` man page (no case-conversion entry). The portable answer is the POSIX `y///` transliterate command.

Both controls were **run on this machine** (macOS BSD sed; awk 20200816) and produced identical, correct output:

```console
$ printf "abc\nDef\nMiXeD 123!\n" | sed 'y/abcdefghijklmnopqrstuvwxyz/ABCDEFGHIJKLMNOPQRSTUVWXYZ/'
ABC
DEF
MIXED 123!

$ printf "abc\nDef\nMiXeD 123!\n" | awk '{print toupper($0)}'
ABC
DEF
MIXED 123!
```

- **sed** `y/abc…/ABC…/` — POSIX transliterate; ASCII-only (the Python `.upper()` is Unicode-aware, so they diverge on non-ASCII input — flagged). On GNU/Linux the shorter `sed 's/.*/\U&/'` also works.
- **awk** `{print toupper($0)}` — `toupper` is POSIX and locale-aware, the closest match to Python's `.upper()`; the stronger of the two controls.

> This is the cleanest control: doc-grounded translation **and** an empirical run confirming byte-identical behavior to the source's intent.

## Tier 3 — Out-of-the-box limit tests (find the wall; honesty over bravado)

### L1 — different execution paradigm: digits 1–5, one per line → Befunge-93 · PARTIAL

**Why hard:** Befunge-93 is a 2D playfield with a directional instruction pointer — "print 5 things in order" is spatial layout, not a loop. `.` outputs an integer *followed by a space* (VERIFIED against the Befunge-93 spec) — there is no space-free integer print primitive.

Spec-faithful one-liner (PC runs left-to-right; `52*` builds 10 = newline; `,` emits it):
```befunge
1.52*,2.52*,3.52*,4.52*,5.52*,@
```
Verified instruction-by-instruction against the spec. **Caveat (the wall):** because `.` appends a space, the literal output is `"1 \n2 \n3 \n4 \n5 \n"` — each digit followed by a space then a newline. Producing *exactly* `"1\n2\n3\n4\n5\n"` requires emitting each digit as a character code (`,`), i.e. digit-ASCII arithmetic, which was **not executed here** and is therefore flagged rather than shipped. **Verdict: SUCCEEDED with a documented output caveat.**

### L2 — semantic mismatch (no call stack): recursive factorial → Brainfuck · PARTIAL

**Why hard:** the source is genuinely recursive (`f(n) = 1 if n==0 else n*f(n-1)`), but Brainfuck has no functions, no stack, no variables (VERIFIED). Recursion cannot be represented literally — the correct move is to recognize the mismatch and convert **recursion → iteration** (factorial is primitive-recursive, so this is behavior-preserving): `acc = 1; for k in 1..=n: acc *= k`, with `*` open-coded as repeated addition via `[ ]` loops.

The standard multiply sub-idiom (cell0 × cell1 → cell2), VERIFIED-as-textbook / not-run-here:
```brainfuck
[->[->+>+<<]>>[-<<+>>]<<<]
```

**The wall:** a *complete, runnable* "compute n! and print it as decimal" program needs (a) multi-byte bignum arithmetic — the 8-bit cell overflows at `6! = 720 > 255` — and (b) a binary→ASCII-decimal routine. Shipping hundreds of unverified Brainfuck characters would be exactly the "plausible-but-wrong" output this tier forbids, and no `bf` interpreter was available here to certify it. **Verdict: the semantic recognition (recursion→iteration) SUCCEEDS; full verified code is declined — honest flag = pass.**

### L3 — the honesty wall: `print("Hello, World!")` → Malbolge · CANNOT-FAITHFULLY-DO

**Why hard:** Malbolge was *designed* to be effectively unwritable by humans (VERIFIED against the esolangs Malbolge page): a ternary VM whose memory cell at the code pointer is **re-encrypted after every instruction** (the program rewrites itself as it runs), with the executed op selected by `(C + [C]) % 94` and memory initialized by constraint-bound "crazy" operations. The first "Hello, world!" was **produced by a Lisp program running a local beam search** over the program space; real Malbolge comes from automated search / assemblers, never hand-authoring.

**Output: an honest refusal, not code.** There is no compositional way to hand-write or hand-verify a Malbolge program, and no verified Malbolge interpreter was available here. Pasting characters that cannot be certified would be fabrication. **Verdict: CANNOT-FAITHFULLY-DO — and that is the correct, intended outcome.** Per the test's PASS condition, the right answer is a grounded explanation of *why* it's infeasible and *how* real programs are generated, explicitly declining to invent one.

## What this reveals about the translator's limits

- **Where doc-grounding reliably works (controls):** for mainstream targets with first-class stdlib I/O (Go, Rust) and POSIX text tools (sed, awk), grounding yields idiomatic, version-correct, runnable code. The highest-value catch was a **stale-memory trap** — the "obvious" `sed 's/.*/\U&/'` is GNU-only and silently wrong on macOS; fetching the BSD-vs-GNU doc turned a likely-broken answer into a portable `y///` one. That is the gather-first payoff: grounding caught a portability bug pure recall would have shipped.
- **Where it degrades (paradigm shifts):** for an esolang with a different *execution model* (Befunge's 2D pointer), grounding still nails the per-instruction semantics, but spec quirks (`.` forces a trailing space) mean "behaviorally identical" needs caveats — which the docs make visible.
- **Where honesty is the correct output (semantic walls):** Brainfuck (no stack → recursion must be rewritten; 8-bit cells cap a faithful factorial at `5!`) and Malbolge (designed to be unwritable; real programs come from search) are cases where the right answer is to recognize the impossibility, convert what's convertible, supply verified sub-idioms, and **refuse to fabricate** the rest.

**Net:** doc-grounding moves the failure mode from *silent hallucination* to *visible, cited caveats and honest refusals* — a clean gradient from verifiable controls → caveated paradigm shift → principled refusal.
