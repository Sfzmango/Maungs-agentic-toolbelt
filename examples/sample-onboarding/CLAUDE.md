## Project overview

ExampleApp is a B2B SaaS helpdesk that lets a business run its customer support inside its own private workspace. Each customer company is an **Organization** (the tenant); its people are **Members** with a role (`owner`, `agent`, or `viewer`) who handle support requests (**Tickets**, which carry **Comments** and move through `open → pending → resolved → closed`). Billing attaches a **Plan** (`free`, `team`, `business`) and issues **Invoices** per billing period. The whole product is multi-tenant: every record belongs to exactly one Organization, and an Organization's data is invisible to every other Organization.

- **Authorization is role-based** via policies (`owner` / `agent` / `viewer`); controllers authorize per action. `(needs confirmation — confirm the authorization library, e.g. Pundit, against the Gemfile.)`

## Gotchas

**Multi-tenant isolation is the #1 bug class — and the cardinal rule (front and center).** Every record is scoped to an Organization. A Member of org A must never read, write, or even *discover* a Ticket, Comment, Invoice, or Member belonging to org B. **Cross-tenant access returns 404, NOT 403** — a 403 leaks the existence of the resource, so the 404 is deliberate and load-bearing. The dominant catastrophic bug is a controller that does `Model.find(params[:id])` on an org-owned table instead of scoping through `current_organization`; every org-scoped action needs a cross-tenant-returns-404 spec.

**Team invitations (EX-218) — token + tenancy conventions to follow for any future token feature:**
- **Digest-at-rest token.** An invitation generates a random token, emails the **raw** value inside the accept link, and persists only a **SHA-256 digest** of it. Lookups hash the incoming token and match on the digest, so a database leak never yields a usable invite link. Never store or log the raw token.
- **Single-use, time-limited.** The token is consumed on acceptance (status moves `pending → accepted`) and expires after **7 days**; a `revoked` or `expired` invitation is no longer acceptable.
- **The accept route is the ONLY unauthenticated surface.** The invitee has no account yet, so the accept route is guarded by the token rather than by a tenant session. Every other invitation action (create / list / revoke / resend) runs inside the normal authenticated, org-scoped controller stack.
- **The org is derived FROM the invitation, never from a request parameter.** Acceptance binds the new Member to the issuing Organization, so an invite can only ever mint a member in the org that issued it — there is no param that could point acceptance at another tenant.
- **Invites are owner-only.** Create, revoke, and resend are authorized to the `owner` role; `agent`/`viewer` cannot invite. Only `agent` and `viewer` roles can be *granted* by invitation (no `owner` invites).

**Don't bypass the pre-commit hook.** lefthook runs RuboCop + lint + asset build; `--no-verify` skips it but CI still runs the full suite, so a bypass only costs a round-trip. The authoritative gate is CI.

**Background mail is async.** Invitation (and other transactional) email is delivered via a background job, so a mail-provider hiccup never blocks the request; **Resend** is the manual recovery affordance. `(needs confirmation — confirm the job/adapter wiring.)`
